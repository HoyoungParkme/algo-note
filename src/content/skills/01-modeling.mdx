<Why>
- 알고리즘을 다 알아도 **문제를 보고 어떤 걸 써야 하는지 판단**하지 못하면 못 푼다
- 같은 문제도 모델링 방식에 따라 BFS/DP/그리디 등으로 갈린다
- 코테 시간 안에 풀려면 **읽자마자 알고리즘 후보를 추리는 직감**이 필요
</Why>

<How>
### 문제를 읽고 나서 반드시 거치는 5단계

1. **N의 범위부터 확인** → 허용 복잡도 판단
   - N≤20이면 완전탐색, N≤10만이면 O(N log N) 이하 필요
   - 후보 알고리즘이 절반으로 줄어듦
2. **핵심 키워드 추출** → 알고리즘 후보 좁히기
   - "최단 거리" → BFS/다익스트라
   - "최소/최대" → DP/그리디/이분탐색
   - "모든 경우" → 완전탐색
   - "연결 여부" → BFS/DFS/유니온파인드
3. **입출력 모양 관찰** → 자료구조 결정
   - 노드·간선 → adj
   - 격자 → 2D 배열 + dx/dy
   - 문자열 → 해시/스택
   - 구간 합 → psum
4. **작은 예시로 손으로 시뮬레이션**
   - 코드 짜기 전에 예제 입력을 손으로 따라가기
   - "이 순서로 처리하면 답이 나오네" → 그게 알고리즘
5. **엣지 케이스 미리 생각**
   - N=1, N=0, 최대값, 음수, 중복, 연결 안 된 그래프 등
   - 풀고 나서 생각하면 늦음
</How>

### 키워드 → 알고리즘 빠른 매핑표

| 키워드 | 후보 알고리즘 |
|:--|:--|
| "최단", "최소 이동", "최소 횟수" | BFS (가중치 없음) / 다익스트라 (있음) / 플로이드 (모든 쌍) |
| "최대/최소 값", "경우의 수" | DP (순서) / 그리디 (탐욕 가능) / 완전탐색 (N 작음) |
| "~가 가능한가?" (Yes/No) | 파라메트릭 서치 = 이분탐색 + check |
| "연결", "그룹", "같은 팀" | BFS/DFS / 유니온-파인드 / MST |
| "순서", "선수조건", "의존성" | 위상 정렬 (+ DP for 시간) |
| "구간 합", "연속 K개" | 누적합 / 투 포인터 / 슬라이딩 윈도우 |
| "조건 만족하는 모든 조합" | 백트래킹 (N ≤ 20) |
| "격자", "미로", "2D 맵" | BFS + dx/dy / DFS flood fill / 시뮬레이션 |
</Template>
